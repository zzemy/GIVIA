# GIVIA 开发路线图 V1

## 1. 目标定义

把当前的「礼物识别 + 文化分析 Demo」升级为可落地的跨境送礼智能产品，形成完整闭环：

1. 识别礼物与用户意图
2. 理解目标国家与收礼人画像
3. 进行文化风险评估
4. 输出可解释推荐、话术、包装和寄送建议
5. 保存历史记录，形成复盘与复用能力

---

## 2. 当前项目现状

当前仓库已经具备：

- Next.js + Tauri 的前端基础框架
- 图片/文本输入后的 AI 识别链路
- 国家选择与基础目标群体输入
- LLM 生成文化分析、包装建议、贺卡文案
- 中英双语主流程页面

当前仓库还缺少：

- 真正的多模态识别模块（CLIP/ResNet/OCR/属性抽取）
- 结构化国家禁忌知识库与知识图谱
- 完整的用户画像标签体系
- 规则引擎 + 推荐引擎的独立实现
- TopN 推荐而非单结果分析
- 历史记录、收藏、场景模板、物流支付助手
- 数据存储、评测、监控、管理后台

结论：当前版本适合答辩演示，不足以支撑“完整产品”叙事。

---

## 3. 版本规划

### P0：可答辩、可演示、可复现的最小闭环

目标：把项目从 Demo 提升为“有数据底座的产品原型”。

#### P0 必做模块

1. 场景模板
- 预设生日、商务拜访、节日、乔迁四类模板
- 一键填充场景、关系、正式程度、预算区间

2. 结构化画像输入
- 新增字段：年龄段、性别、职业、关系、场合、预算、送礼目的
- 将当前自由文本 `audience.profile` 拆成结构化字段 + 备注

3. 国家禁忌规则库
- 先做规则库，不强行一步到位做知识图谱
- 首批覆盖 8 个高频国家/地区：`中国、日本、美国、英国、法国、德国、沙特、阿联酋`
- 每个国家至少包含：数字禁忌、颜色禁忌、象征物禁忌、节日注意事项、商务送礼注意事项

4. 风险评估引擎 V1
- 规则引擎打基础分
- LLM 补充语义解释与缓解建议
- 输出统一风险分：`0-100`
- 输出命中规则列表，保证可解释性

5. 推荐引擎 V1
- 先不做 Wide & Deep
- 用“规则过滤 + 标签召回 + 排序打分”替代
- 输出 Top3 礼物建议
- 每条建议附带：推荐理由、风险说明、包装建议、送礼话术

6. 历史记录与收藏
- 保存分析记录
- 收藏推荐礼物
- 支持再次打开历史结果复盘

#### P0 不做

- 自研训练 CLIP / ResNet50
- 协同过滤在线学习
- Wide & Deep 训练管线
- 国际物流、汇率、支付真实接口
- 多语言全量翻译后台

#### P0 产出标准

- 用户可完成一次完整送礼分析闭环
- 风险结果不再完全依赖 LLM 自由发挥
- 能展示 Top3 推荐而不是只有单份分析
- 结果可保存、复看、收藏

---

### P1：产品化增强

目标：增强识别能力、推荐质量和跨境实用性。

#### P1 模块

1. 多模态识别升级
- 接入 OCR，抽取包装文字、品牌名、规格词
- 图像识别结果增加结构化属性：
  - 品类
  - 材质
  - 风格
  - 颜色
  - 品牌/包装文字
- 文本理解模块把“黑色金属笔身、商务风”解析成标签

2. 一键润色升级
- 从当前本地文本清洗升级为 LLM 润色服务
- 目标是把模糊输入变成可用于推荐的标准描述

3. 推荐引擎 V2
- 引入协同过滤或相似标签召回
- 增加基于场景、预算、国家的重排序策略

4. 物流与汇率助手
- 汇率 API
- 预估运费、清关提醒、时效说明
- 输出寄送注意事项

5. 多语言扩展
- 主流程真正支持 `zh/en/ja/fr`
- 分离界面翻译与模型输出语言

---

### P2：算法与商业化增强

目标：形成比赛中的“技术创新点”和后续商业化基础。

#### P2 模块

1. 跨文化知识图谱
- 实体：国家、节日、禁忌、礼物、收礼人群体、场景
- 关系：适宜、禁忌、弱禁忌、商务推荐、节日推荐、替代建议

2. 风险评估引擎 V2
- 规则库 + 知识图谱 + LLM 推理联合
- 输出风险来源权重
- 支持“为什么不适合”和“如何降风险”的解释链

3. 推荐引擎 V3
- Wide & Deep 或双塔召回 + 排序
- 将用户画像、国家规则、礼物向量、历史行为统一建模

4. B 端开放能力
- 文化合规校验 API
- 企业礼品平台嵌入式分析接口

5. C 端会员化
- 无限次分析
- 高级国家规则包
- 专属顾问建议

## 6. API 设计建议

### P0 API

#### `POST /api/recognition/parse`

用途：
- 接收图片或文本
- 输出标准化礼物结构

返回建议：

```json
{
  "gift": {
    "name": "钢笔",
    "category": "pen",
    "materials": ["metal"],
    "styles": ["business"],
    "colors": ["black"],
    "brand_text": ["Parker"]
  },
  "source": "vision+text",
  "confidence": 0.84
}
```

#### `POST /api/risk/evaluate`

用途：
- 输入国家、画像、礼物标签
- 返回规则命中与综合风险分

#### `POST /api/recommend/top`

用途：
- 输出 Top3 推荐结果

#### `POST /api/analysis/run`

用途：
- 编排完整流程：
  - 识别
  - 风险评估
  - 推荐
  - 解释生成
  - 落历史记录

#### `GET /api/history`

用途：
- 获取历史分析记录

#### `POST /api/favorites`

用途：
- 收藏推荐礼物

## 7. 算法方案建议

### 7.1 识别模块

#### P0

- 图像识别：继续使用现有视觉模型接口
- 文本理解：继续使用 LLM
- 输出统一结构化标签，不再只输出 `label + description`

#### P1

- OCR：识别包装文字、品牌名、数字信息
- 图像属性抽取：颜色、材质、风格、包装形态

#### P2

- 如果需要比赛包装，可说明：
  - 图像 backbone 可采用 CLIP/ResNet50
  - OCR 补充品牌和文字信息
  - LLM 负责语义融合与标签标准化

说明：P2 可以写进答辩方案，但工程上不建议在当前仓库直接自研训练。

### 7.2 风险评估引擎

#### P0 公式

```text
总风险分 = 规则命中分 * 0.7 + LLM语义修正分 * 0.3
```

其中：

- 规则命中分：根据国家规则库直接计算
- LLM语义修正分：判断语义联想、礼貌程度、场景匹配

输出：

- `risk_score`
- `risk_level`
- `hit_rules`
- `explanation`
- `mitigation_actions`

#### P2 升级

- 引入知识图谱推理
- 把规则命中拆成多维贡献值

### 7.3 推荐引擎

#### P0

先用三段式：

1. 规则过滤
- 剔除禁忌品类和高风险礼物

2. 标签召回
- 按国家、场景、预算、职业、关系匹配候选礼物

3. 排序打分
- 综合预算匹配、场景匹配、风险倒扣、风格匹配

#### P1

- 加入协同过滤或相似用户偏好

#### P2

- Wide & Deep
- Wide：预算、场景、职业、国家规则
- Deep：礼物特征向量、文化禁忌向量、用户历史偏好

## 12. 最终建议

当前最优策略不是马上追求“所有高级算法一次到位”，而是：

1. 先把规则库、画像、推荐和历史记录做实
2. 再把 OCR、多语言、物流助手补齐
3. 最后再包装知识图谱、Wide & Deep、多模态融合为技术创新升级路线

这样做的好处：

- 比赛答辩时叙事完整
- 开发工作量可控
- 结果可复现，不会完全依赖模型随机输出
- 后续容易扩成 B 端 API 或 C 端会员产品

---

## 13. 建议的下一步开发顺序

严格按下面顺序推进：

1. 重构数据结构与 API 边界
2. 落地国家规则库
3. 落地结构化画像与场景模板
4. 落地风险评估引擎 V1
5. 落地 Top3 推荐引擎 V1
6. 落地历史记录与收藏
7. 再做 OCR、汇率、物流、多语言扩展

如果资源有限，P0 做完就已经足够支撑“比赛版 + 原型版产品”。
